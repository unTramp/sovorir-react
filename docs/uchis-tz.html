Техническое задание

Проект: Учись!
Тип продукта: веб-платформа для изучения армянского языка
Формат документа: финальное ТЗ на основе анализа текущего кода и уточнений заказчика

1. Общая информация

1.1. Назначение системы

Платформа Учись! предназначена для изучения армянского языка русскоязычными пользователями: начинающими, учениками среднего уровня, представителями диаспоры и релокантами.

Система должна помогать пользователю:

  • свободно разговаривать на армянском языке;
  • читать на армянском языке;
  • понимать армянскую речь и учебный контент.

1.2. Формат продукта

Система реализуется как web-приложение с адаптацией под:

  • мобильные устройства;
  • desktop-браузеры.

На первом этапе основной формат поставки:

  • mobile web;
  • desktop web;
  • PWA с ограниченной офлайн-поддержкой.

2. Пользовательские роли

В системе должны быть предусмотрены следующие роли:

  • Ученик
  • Учитель
  • Администратор

2.1. Ученик

Ученик должен иметь доступ к:

  • образовательному контенту;
  • урокам и уровням;
  • словарю;
  • упражнениям и тестам;
  • записи на онлайн-уроки;
  • просмотру своего прогресса, статистики и достижений;
  • подписке и управлению доступом в рамках своего тарифа.

2.2. Учитель

Учитель должен иметь возможность:

  • вести онлайн-уроки для пользователей с PRO-подпиской;
  • просматривать работы и записи учеников;
  • давать комментарии и разбор ошибок;
  • корректировать учебные ошибки;
  • публиковать и редактировать контент через отдельный веб-интерфейс, если ему назначены административные функции.

2.3. Администратор

Администратор должен иметь доступ к отдельной веб-панели и управлять:

  • пользователями;
  • уроками;
  • словарем;
  • тестами и практикой;
  • live lessons;
  • подписками;
  • аналитикой;
  • уведомлениями;
  • общим контентом платформы.

Ограничение:

  • отдельной гибкой системы прав доступа между администратором и учителем на текущем этапе не требуется;
  • панель управления может использоваться администратором и учителем по организационной модели проекта.

3. Бизнес-логика обучения

3.1. Учебная модель

Обучение должно быть построено как структурированная последовательная программа.

Основные принципы:

  • контент разбит на уровни;
  • внутри уровней доступны уроки;
  • внутри уроков доступны блоки контента;
  • прохождение должно быть поэтапным;
  • будущие уроки не должны быть доступны до выполнения условий открытия.

3.2. Доступ к урокам

Система должна:

  • открывать пользователю только доступные уроки текущего этапа;
  • блокировать будущие уроки;
  • визуально показывать статус урока: доступен, текущий, завершен, заблокирован.

3.3. Условие завершения урока

Урок считается завершенным, если:

  • пользователь прошел все обязательные блоки урока;
  • пользователь успешно завершил итоговый тест по уроку.

3.4. Домашние задания и дедлайны

На текущем этапе:

  • домашние задания не требуются;
  • дедлайны не требуются;
  • рекомендации следующего урока отдельно не требуются, кроме логики последовательного доступа.

4. Авторизация и учетная запись

4.1. Регистрация и вход

Система должна поддерживать регистрацию и вход пользователя. На текущем этапе базовым сценарием следует считать:

  • вход/регистрация по email;
  • подтверждение через код, отправленный на email.

Допускается, что точная схема будет уточнена на этапе проектирования backend и UX.

4.2. Восстановление доступа

Система должна поддерживать:

  • восстановление доступа;
  • подтверждение email;
  • удаление аккаунта по запросу пользователя.

4.3. Профиль пользователя

Минимальные данные профиля:

  • имя;
  • контактный способ связи;
  • телефон и/или социальная сеть.

На этапе UX-проектирования требуется финально зафиксировать, какие поля обязательны, а какие опциональны.

5. Функциональные модули ученической части

5.1. Главная навигация

Пользовательская часть должна включать следующие разделы:

  • главная страница;
  • уроки;
  • практика;
  • словарь;
  • заметки/материалы;
  • live lessons;
  • статистика;
  • настройки;
  • профиль;
  • подписка.

Навигация должна работать без полной перезагрузки страницы.

5.2. Уроки

Система должна отображать:

  • список уровней;
  • список уроков внутри уровня;
  • статус прохождения;
  • прогресс по каждому уроку.

Каждый урок должен поддерживать следующие типы блоков:

  • текстовые блоки;
  • фразы;
  • аудио;
  • запись ответа;
  • правила;
  • тестовые блоки.

На текущем этапе дополнительные типы блоков не обязательны.

5.3. Контент урока

Контент урока должен отображаться пошагово и быть удобным для мобильного прохождения.

Система должна поддерживать:

  • последовательный просмотр блоков;
  • отметку о прохождении блока;
  • навигацию между страницами урока;
  • отображение статуса выполнения;
  • итоговое завершение урока после прохождения всех блоков и теста.

5.4. Аудио и произношение

Платформа должна поддерживать:

  • прослушивание учебных аудиоматериалов;
  • запись голоса пользователя;
  • воспроизведение собственной записи;
  • хранение записей на сервере;
  • привязку записи к конкретному уроку, блоку и пользователю.

На втором этапе или как отдельный функциональный модуль должна быть реализована:

  • AI-оценка произношения через отдельные model/API-запросы.

Дополнительно учитель должен иметь возможность:

  • прослушивать записи ученика;
  • оставлять комментарии;
  • давать обратную связь по произношению и ошибкам.

5.5. Практика

Система должна поддерживать отдельный режим практики с упражнениями.

Обязательные режимы:

  • карточки;
  • matching;
  • multiple choice;
  • open answer;
  • speaking drills.

На текущем этапе диктант не требуется.

5.6. Тестирование

Система должна поддерживать тесты:

  • внутри уроков;
  • как итоговый тест урока.

Требования:

  • тесты задаются вручную через админ-панель;
  • результаты тестов сохраняются;
  • при неуспешном прохождении система должна поддерживать:
    • повтор;
    • подсказки;
    • повторное прохождение;
    • в перспективе комментарии и разбор с преподавателем.

Дополнение на будущее:

  • должна быть предусмотрена архитектурная возможность автоматической генерации тестов, но не обязательна для MVP.

5.7. Словарь

Словарь должен быть общим для всех пользователей курса.

Функции словаря:

  • поиск;
  • фильтрация;
  • группировка;
  • деление по урокам и темам;
  • добавление в избранное;
  • добавление в список “на повторение”.

Словарь должен использоваться как источник для практики и повторения.

5.8. Заметки и материалы

Заметки преподавателя в ученической части должны быть доступны только для чтения.

Пользователь должен иметь возможность:

  • просматривать заметки;
  • экспортировать материалы.

На текущем этапе:

  • загрузка файлов пользователем не требуется;
  • вложения и ссылки в пользовательских заметках не требуются;
  • личные заметки ученика не обязательны.

5.9. Live Lessons

Система должна поддерживать полноценную запись на онлайн-уроки.

Требования:

  • отображение расписания;
  • запись на занятие;
  • указание времени и формата связи;
  • лимиты мест;
  • лист ожидания;
  • напоминания;
  • история прошедших занятий;
  • информация о персональных уроках.

Интеграция видеосвязи:

  • звонок будет проходить через WhatsApp;
  • отдельные интеграции с Zoom/Meet не требуются.

Записи эфиров:

  • не требуются.

5.10. Статистика и прогресс

Система должна сохранять и отображать пользователю:

  • streak;
  • завершенные уроки;
  • время в обучении;
  • процент правильных ответов;
  • количество выученных слов;
  • активность по аудио и произношению;
  • общий учебный прогресс;
  • достижения и бейджи.

Прогресс должен синхронизироваться между устройствами.

5.11. Подписка

Система должна поддерживать модель монетизации:

  • бесплатный пробный период;
  • платная подписка PRO.

Логика доступа:

  • часть функций доступна в бесплатный период;
  • расширенные функции, включая live lessons с учителем, доступны по PRO.

Платежная интеграция:

  • желательна;
  • конкретный платежный провайдер будет определен на этапе проектирования.

5.12. Уведомления

Система должна поддерживать уведомления.

Приоритетные события:

  • персональный урок;
  • подписка.

Каналы уведомлений:

  • email;
  • push.

Допускается расширение на следующих этапах.

6. Административная и учительская панель

Система должна включать отдельный web-интерфейс для управления платформой.

6.1. Управление пользователями

Админ-панель должна позволять:

  • просматривать пользователей;
  • редактировать профильные данные;
  • управлять статусом подписки;
  • просматривать прогресс;
  • просматривать записи и результаты;
  • при необходимости блокировать или деактивировать аккаунт.

6.2. Управление контентом

Админ-панель должна позволять:

  • создавать и редактировать уровни;
  • создавать и редактировать уроки;
  • управлять порядком уроков;
  • задавать правила открытия уроков;
  • редактировать блоки уроков;
  • создавать и редактировать словарь;
  • создавать и редактировать тесты;
  • создавать и редактировать материалы и заметки;
  • управлять live lessons и расписанием.

6.3. Управление образовательной обратной связью

Панель должна позволять учителю/администратору:

  • просматривать записи учеников;
  • оставлять комментарии;
  • вести разбор ошибок;
  • комментировать результат урока или упражнения.

6.4. Аналитика

Админ-панель должна предоставлять:

  • аналитику по пользователям;
  • аналитику по урокам;
  • аналитику по подпискам;
  • аналитику по активности;
  • аналитику по тестам и практике.

6.5. Права доступа

На текущем этапе отдельная сложная ролевая модель в админ-панели не требуется. Достаточно единой административной панели для пользователей с административным доступом.

7. Backend и API

Backend на текущий момент отсутствует и должен быть спроектирован.

Серверная часть должна обеспечивать:

  • аутентификацию и авторизацию;
  • хранение пользовательских профилей;
  • хранение прогресса;
  • хранение результатов тестов и практики;
  • хранение аудиозаписей;
  • хранение словаря и учебного контента;
  • хранение расписания live lessons;
  • управление подписками;
  • уведомления;
  • API для AI-оценки произношения;
  • API для административной панели.

7.1. Основные серверные сущности

Необходимо предусмотреть сущности:

  • user;
  • role;
  • subscription;
  • level;
  • lesson;
  • lesson block;
  • quiz;
  • quiz result;
  • dictionary item;
  • recording;
  • teacher comment;
  • live lesson;
  • enrollment/booking;
  • notification;
  • achievement;
  • progress.

7.2. Хранение файлов

Аудиозаписи и медиафайлы должны храниться на сервере. Допускается использование объектного хранилища.

8. Офлайн-режим

Система должна поддерживать ограниченный офлайн-режим.

Требования:

  • офлайн доступны только уже загруженные материалы;
  • пользователь может просматривать ранее загруженные уроки и материалы;
  • переход к новым материалам допускается только при наличии интернета;
  • приложение должно быть оптимизировано по весу;
  • кэширование должно быть управляемым и не приводить к избыточному потреблению памяти устройства.

9. Персональные данные и безопасность

Требования по данным:

  • не хранить лишние персональные данные;
  • хранить только минимально необходимый набор данных;
  • обеспечить безопасную обработку данных пользователя.

Дополнительные требования:

  • предусмотреть резервное копирование для восстановления в крайнем случае;
  • реализовать безопасную авторизацию;
  • обеспечить защиту пользовательских записей и прогресса;
  • учесть удаление аккаунта и данных по запросу пользователя.

На этапе архитектуры также требуется определить:

  • политику хранения данных;
  • сроки хранения;
  • механизм резервного восстановления;
  • политику доступа к аудиозаписям.

10. UX и дизайн

На текущем этапе готового брендбука нет. Дизайн-система должна быть сформирована на основе текущего продукта и кода с дальнейшей доработкой.

Рекомендуемое позиционирование интерфейса:

  • дружелюбный;
  • современный;
  • понятный;
  • ориентированный на обучение;
  • аккуратный и уверенный, без перегрузки.

UX-принципы:

  • mobile-first;
  • простая пошаговая навигация;
  • заметный прогресс;
  • акцент на аудио-сценарии;
  • минимальное количество действий для прохождения урока.

10.1. Accessibility

Нужно предусмотреть базовую доступность, особенно для сценариев с микрофоном:

  • понятные запросы разрешения на микрофон;
  • ясные состояния записи, ожидания, ошибки и воспроизведения;
  • удобное использование на мобильных устройствах;
  • достаточную контрастность и размер интерактивных элементов.

11. Нефункциональные требования

Система должна:

  • корректно работать в современных мобильных и desktop-браузерах;
  • иметь адаптивный интерфейс;
  • поддерживать SPA-модель;
  • поддерживать PWA;
  • обеспечивать приемлемую скорость загрузки;
  • быть оптимизированной по размеру клиентского приложения;
  • обеспечивать устойчивую работу при нестабильной сети;
  • поддерживать синхронизацию прогресса между устройствами;
  • быть готовой к расширению функциональности.

12. MVP

В MVP рекомендуется включить следующий объем работ.

12.1. Ученическая часть

  • регистрация и вход по email с подтверждением;
  • профиль пользователя;
  • уровни и уроки с поэтапным открытием;
  • отображение блоков урока текущих типов;
  • прохождение уроков и фиксация прогресса;
  • итоговые тесты уроков;
  • словарь с поиском, фильтрами и избранным;
  • практика: карточки, matching, multiple choice, open answer, speaking drills;
  • запись голоса пользователя;
  • загрузка записей на сервер;
  • статистика прогресса;
  • достижения и бейджи в базовом виде;
  • просмотр и запись на live lessons;
  • free trial и PRO-подписка;
  • уведомления о персональном уроке и подписке;
  • адаптивная web-версия;
  • ограниченный офлайн для уже загруженного контента.

12.2. Админ/учительская часть

  • вход в административную панель;
  • CRUD для уровней и уроков;
  • CRUD для блоков урока;
  • CRUD для словаря;
  • CRUD для тестов;
  • CRUD для live lessons;
  • просмотр пользователей и их прогресса;
  • просмотр ученических записей;
  • базовые комментарии преподавателя;
  • базовая аналитика;
  • управление подпиской пользователя.

13. Второй этап

Во второй этап рекомендуется вынести:

  • AI-оценку произношения;
  • расширенную обратную связь преподавателя по записям;
  • разбор ошибок внутри уроков;
  • автоматическую генерацию тестов;
  • более продвинутую аналитику;
  • более сложную систему уведомлений;
  • улучшенную систему достижений и геймификации;
  • расширенное резервное восстановление;
  • углубленную дизайн-систему и бренд-гайд;
  • более тонкую административную ролевую модель, если она потребуется.

14. Открытые точки для проектирования

Следующие вопросы нужно зафиксировать на этапе системного проектирования и UX:

  • точный сценарий регистрации и входа;
  • точный состав полей профиля;
  • правила бесплатного периода;
  • ограничения PRO-доступа;
  • выбор платежного провайдера;
  • формат напоминаний и частота уведомлений;
  • логика лимитов и листа ожидания для live lessons;
  • формат teacher feedback;
  • архитектура AI-проверки произношения;
  • хранилище файлов и стратегия резервного копирования.

15. Итог

Проект Учись! должен стать образовательной web-платформой для изучения армянского языка с поэтапным обучением, сильным акцентом на аудирование и произношение, синхронизацией прогресса, подписочной моделью и отдельной административной средой для управления контентом и учениками.
