# Текущий функционал админки Sovorir

Сейчас админка представляет собой отдельное локальное приложение для редактирования только `Урока 1`, а не полную административную систему всей платформы. Отдельное приложение находится в `admin-app`, а основной экран конструктора расположен в `AdminLessonBuilderView.tsx`.

## 1. Вход и доступ

Админка умеет:

- открывать отдельный интерфейс для ролей `teacher` и `admin`
- показывать экран логина, если пользователь не авторизован
- блокировать доступ для пользователей с неподходящей ролью

Точка входа:

- `admin-app/src/AdminApp.tsx`

## 2. Формат панели

Сейчас это `MVP lesson builder`, сфокусированный на одном уроке:

- редактируется только `Урок 1: Приветствия и прощания`
- нет общего каталога уроков
- нет отдельного раздела пользователей, аналитики, подписок, словаря или live lessons

То есть текущая админка решает одну конкретную задачу:

- собрать и редактировать структуру первого урока

## 3. Модель данных

Урок строится по модели:

- `Lesson`
- `Section`
- `Block`

Эта модель зафиксирована в:

- `docs/content-model.md`

## 4. Что можно редактировать

### 4.1. Метаданные урока

Сейчас можно менять:

- название урока
- `slug`
- описание
- статус урока:
  - `draft`
  - `published`
  - `archived`

### 4.2. Секции урока

Для секций поддерживается:

- добавление секции
- удаление секции
- изменение названия секции
- изменение типа секции
- изменение порядка секций
- редактирование дополнительного JSON секции

### 4.3. Блоки внутри секции

Для блоков поддерживается:

- добавление блока в секцию
- вставка блока ниже существующего блока
- удаление блока
- изменение порядка блока вверх/вниз
- изменение типа блока
- редактирование содержимого блока через JSON

## 5. Поддерживаемые типы блоков

Сейчас в интерфейсе преподавателю показываются семантические типы:

- `heading` — внутренний заголовок
- `text` — поясняющий текст
- `teacherBubble` — бабл преподавателя
- `phraseCard` — карточка фразы
- `rule` — правило
- `readingText` — текст для чтения
- `video` — видео

Это определено в:

- `src/lib/semanticContent.ts`

Также зарезервированы, но пока не используются:

- `studentBubble`
- `pronunciationPrompt`

## 6. Как редактируется контент

Сейчас блоки редактируются через JSON, а не через специальные формы для каждого типа блока.

То есть для каждого блока можно:

- выбрать тип
- увидеть, какой внутренний legacy-тип хранится
- руками поменять JSON блока
- сохранить изменения

Этот подход удобен для разработки и тестирования модели контента, но пока ещё не является идеальным UX для преподавателя.

## 7. Синхронизация с ученическим приложением

Сейчас локально админка и ученическое приложение синхронизированы.

Это работает так:

- админка и ученическое приложение запускаются под одним origin
- админка сохраняет данные урока в `localStorage`
- ученическое приложение читает тот же lesson source
- после сохранения в админке ученик может увидеть изменения в локальном режиме

Ключевой общий слой:

- `src/lib/adminLessonBuilderStorage.ts`

Локальные адреса:

- ученическое приложение: `http://172.20.10.10:4183/`
- админка: `http://172.20.10.10:4183/admin/`

## 8. Что специально убрано из текущего MVP

Для упрощения первого этапа из админки временно убраны:

- AI-генерация урока
- AI-review урока
- AI-генерация квиза
- список всех уроков
- список курсов
- создание новых уроков
- удаление уроков из общего каталога
- широкая CMS-логика для всей платформы

То есть текущая версия — это сознательно урезанный `lesson editor` под первый урок.

## 9. Как работает слой данных

Админка умеет работать в двух режимах:

- `mock/localStorage`
- `API/backend`

Но в текущем локальном MVP-цикле используется именно локальный режим, чтобы:

- быстро править урок без деплоя
- не зависеть от backend
- сразу видеть результат в ученическом приложении

Репозиторий данных:

- `src/lib/adminLessonBuilderRepository.ts`

## 10. Что пока не реализовано

Сейчас в админке ещё нет:

- визуального preview блока в формате ученика
- специальных форм по типам блоков вместо JSON
- отдельного управления словарём
- отдельного управления квизами как сущностью
- версионирования уроков
- workflow `draft -> review -> publish`
- медиа-библиотеки
- аналитики
- раздела пользователей
- подписок
- live lessons

## 11. Краткий итог

На текущий момент админка — это:

- отдельное приложение
- доступное только ролям `teacher/admin`
- минимальный визуальный конструктор урока
- заточенный под `Урок 1`
- работающий с секциями и блоками
- синхронизированный локально с ученическим приложением
- без AI и без лишних административных модулей

То есть это уже не просто черновой экран, а рабочий MVP-конструктор первого урока, который можно использовать как основу для дальнейшего развития полноценной admin-панели.
